# FreePiano 路径分类说明（2026-03）

更新时间：2026-03-24

---

# 一、文档目的

本说明用于解释 FreePiano 当前工程中“相对路径文件”是如何分类与定位的，重点回答以下问题：

* 哪些文件属于运行时配置文件
* 哪些文件属于历史资源 / 数据文件
* `config_get_media_path()` 当前如何处理不同类型的路径
* 为什么 `freepiano.cfg` 现在和其他相对资源走不同规则

---

# 二、核心结论

当前项目中的相对路径文件，可以先按用途分为两大类：

## 2.1 运行时配置文件

典型代表：

* `freepiano.cfg`

特点：

* 程序启动时读取
* 程序退出时保存
* 记录用户当前配置状态
* 属于“运行实例相关文件”
* 更适合放在 `freepiano.exe` 同目录

当前规则：

> `freepiano.cfg` 默认放在 **exe 同目录**

例如在 x64 Debug 构建下：

```txt
E:\Source\repos\freepiano\vc\x64\Debug\freepiano.exe
E:\Source\repos\freepiano\vc\x64\Debug\freepiano.cfg
```

---

## 2.2 历史资源 / 数据文件

典型代表：

* keymap 文件
* 某些默认数据文件
* 部分通过相对路径读取的资源文件
* 历史工程中统一放在 `data/` 或类似目录的文件

特点：

* 更偏“读取型资源”
* 历史上往往不是每次退出都要回写
* Debug 构建通常依赖固定工程目录结构来查找
* 属于“工程资源文件”，而不是“运行实例配置文件”

当前规则：

* Debug：优先按历史 `data` 目录规则处理
* Release：按 exe 同目录相对路径处理

---

# 三、当前路径解析函数

当前项目的核心路径入口之一是：

```cpp
config_get_media_path(char *buff, int buff_size, const char *path)
```

它的基本行为是：

## 3.1 如果传入的是绝对路径

例如：

```txt
D:\Music\demo.fpm
```

则：

* 直接使用该路径
* 不做额外 media/data 重定位

---

## 3.2 如果传入的是相对路径

例如：

```txt
freepiano.cfg
some.map
plugins\abc.dll
```

则会进入分类逻辑。

---

# 四、`freepiano.cfg` 的特殊处理

## 4.1 为什么要特殊处理？

历史上，`config_get_media_path()` 在 Debug 下会把相对路径统一映射到：

```txt
..\..\data\
```

这对于“资源文件”是合理的，
但对于 `freepiano.cfg` 这样的运行时配置文件就不合理。

你已经实际验证过，旧行为会导致日志显示：

```txt
E:\Source\repos\freepiano\vc\data\freepiano.cfg
```

这会带来两个问题：

1. 配置文件位置不符合直觉
2. 很容易因为目标目录不存在而导致保存失败

---

## 4.2 当前修复后的规则

现在代码里会先判断：

```cpp
is_runtime_config_path(path)
```

如果识别为：

```txt
freepiano.cfg
```

则：

* 无论 Debug / Release
* 都使用 exe 同目录

即：

```txt
<freepiano.exe 所在目录>\freepiano.cfg
```

---

# 五、历史资源 / 数据文件的处理规则

对于除 `freepiano.cfg` 之外的其他相对路径，当前仍保留历史规则：

## 5.1 Debug 构建

会把基目录定位到：

```txt
..\..\data\
```

也就是从输出目录回到工程中的 data 目录。

这样做的目的，是兼容历史工程的调试方式：

* 可执行文件在 `vc\x64\Debug\`
* 资源数据在工程固定目录中
* 通过相对跳转去查找这些资源

---

## 5.2 Release 构建

则更多按 exe 同目录处理：

```txt
.\
```

也就是：

* 相对资源与最终发布目录一起部署

---

# 六、当前可见的路径分类示例

以下示例基于当前代码与调用点整理。

## 6.1 配置文件

### `freepiano.cfg`

用途：

* 用户运行时配置

当前默认路径：

```txt
<exe目录>\freepiano.cfg
```

典型调用点：

* `config_preload_ui_language("freepiano.cfg")`
* `config_load("freepiano.cfg")`
* `config_save("freepiano.cfg")`

---

## 6.2 keymap / 脚本类文件

用途：

* 按键映射数据
* 历史配置脚本输入

当前分类：

* 更接近“历史资源 / 数据文件”

说明：

* 若使用相对路径，Debug 下仍可能走历史 `data/` 定位规则
* 不属于“运行实例配置文件”那一类

---

## 6.3 插件/其他相对文件

用途：

* 某些通过相对路径传入的资源文件或插件路径

当前分类：

* 一般仍视为资源型路径，而不是配置文件

说明：

* 如果调用点传入的是绝对路径，则直接使用
* 如果传入的是相对路径，则仍按历史 media/data 规则处理

---

# 七、为什么当前不把所有相对路径都改成 exe 同目录？

原因很简单：

> 风险过高。

如果一次性把所有相对路径都改成 exe 同目录，可能会导致：

* Debug 下找不到历史 data 目录中的资源
* keymap / 默认数据 / 其他依赖文件全部换位置
* 引发更多回归问题

因此当前采取的策略是：

## 7.1 先修明确不合理的部分

例如：

* `freepiano.cfg`

这是明显应该与 exe 同目录的运行时配置文件，所以优先修正。

## 7.2 对其他相对资源保持历史兼容

这样可以：

* 降低调试环境回归风险
* 避免一次性破坏工程资源布局

---

# 八、当前维护建议

## 8.1 新增文件时先明确分类

以后如果新增相对路径文件，建议先判断它属于哪类：

### 属于运行实例文件

例如：

* 用户配置
* 临时状态
* 本次运行要保存的文件

建议：

* 放在 exe 同目录
* 或单独设计明确的用户数据目录规则

### 属于资源 / 数据文件

例如：

* 默认模板
* 语言资源
* 历史脚本数据
* 只读输入资源

建议：

* 继续走资源定位规则
* 或在后续重构时统一资源根目录策略

---

## 8.2 不要继续把两类文件混用一个语义

虽然当前仍共用 `config_get_media_path()`，但语义上已经要区分：

* 运行时配置文件
* 历史资源文件

如果后续继续演进，建议最终把它们拆成：

* `config_get_runtime_path(...)`
* `config_get_resource_path(...)`

不过这属于后续结构性整理，不是当前阶段必须立即做的事。

---

# 九、一句话总结

> 当前项目中，`freepiano.cfg` 属于“运行时配置文件”，默认应放在 `freepiano.exe` 同目录；而其他大多数相对路径文件仍属于“历史资源 / 数据文件”，在 Debug 下继续沿用 `data/` 目录定位规则，以保持工程兼容性。
