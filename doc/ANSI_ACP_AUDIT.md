# FreePiano ANSI / ACP 遗留点审计（2026-03）

更新时间：2026-03-24

---

# 一、审计目标

本清单用于识别项目中仍然存在的：

* `CP_ACP` 编码转换
* `A` 版 Win32 API
* 窄字符 UI 文本更新
* 历史 ANSI 兼容链路

原则：

* **不做一次性大重构**
* **优先处理高风险、用户可见、与路径/菜单/UI 相关的链路**
* **对难以立即替换的历史路径先做集中封装、日志与风险标记**

---

# 二、当前发现的主要遗留点

## 2.1 高优先级

### A. 文件对话框路径转换链路

位置：

* `src/gui.cpp`

现象：

* `OPENFILENAMEW` 已使用宽字符 API
* 但初始目录与对话框返回路径仍需在 `char*` 与 `wchar_t*` 之间转换
* 目前底层仍依赖 `CP_ACP`

风险：

* 遇到无法在 ACP 中表达的路径时，转换会失败
* 这类问题对非 ASCII 路径最敏感

当前状态：

* 已新增集中转换辅助函数
* 已在转换失败时输出日志
* 已去掉 silent fail

后续方向：

* 推动更多文件路径链路改为宽字符或 UTF-8 内部表示

---

### B. 菜单文本 fallback 到 `AppendMenuA`

位置：

* `src/gui.cpp`

现象：

* 过去菜单文本在宽字符转换失败时，会回退到 `AppendMenuA`

风险：

* 会把本应统一的 W 菜单链路拉回 ANSI
* 增加 UI 行为不一致与乱码风险

当前状态：

* 已移除 `AppendMenuA` fallback
* 统一改为 `AppendMenuW`
* 转换失败时输出 warning 日志并避免继续走 A API

---

## 2.2 中优先级

### C. `SetWindowTextA` / 窄字符编辑框更新

位置示例：

* `src/gui.cpp`

现象：

* 历史上若干编辑框、脚本文本框曾使用 `SetWindowTextA`

风险：

* 对非 ASCII 内容有潜在限制
* 与 Unicode 主链路不完全一致

当前状态：

* `gui.cpp` 中用户可见文本的直接 `SetWindowTextA` / `SetDlgItemTextA` / `GetWindowTextA` / `GetDlgItemTextA` 调用已基本清空
* 已改为统一封装后走 W 路径

后续建议：

* 继续关注其他模块是否仍存在同类直接 A 版文本 API

---

### D. `LoadStringA` 旧资源读取

位置：

* `src/language.cpp`

现象：

* `lang_load_string_array()` 仍使用 `LoadStringA`

风险：

* 属于历史兼容接口
* 与当前 JSON i18n 主链路不一致

当前状态：

* 已确认项目内暂无直接调用点
* 已增加首次调用 warning 日志

建议：

* 如果后续仍无调用，可考虑进一步降级为明确的 legacy-only 接口

---

### E. `CP_ACP` 普通文本转换

位置示例：

* `src/gui.cpp`
* `src/output_asio.cpp`
* `src/synthesizer_vst.cpp`
* `src/language.cpp`
* `src/config.cpp`

现象：

* 仍有若干 ACP 文本转换

风险：

* 在插件名、设备名、外部路径等场景下可能存在显示或转换损失

建议：

* 逐步把“路径/显示文本/插件名”分类治理
* 优先改与用户可见内容相关的部分

---

## 2.3 低优先级

### F. 调试输出 / 历史窄字符格式化

位置示例：

* 若干 `_snprintf`
* 若干历史调试代码

说明：

* 这类本身不一定是编码问题
* 需要与“真正的 ANSI UI / 路径链路”区分开来

# 四、当前边界收敛策略

对于仍然无法立即移除的 ANSI / ACP 路径，当前统一采用以下策略：

1. **明确标记为兼容边界（legacy boundary）**
   * 在代码附近注明该转换存在的历史原因
2. **集中封装或集中出现**
   * 避免同类转换散落到业务逻辑中
3. **失败可观测**
   * 转换失败时必须有 warning / error 日志
4. **优先保护用户可见行为**
   * 失败时给出安全兜底，而不是静默乱码或继续走 A API

说明：

* 这不是“接受长期保留问题”
* 而是让剩余兼容边界先变得可理解、可定位、可逐步替换

# 五、当前阶段收口结论

## 5.1 已完成的关键收口

* 文件对话框路径转换入口已集中封装
* 菜单文本不再回退到 `AppendMenuA`
* `gui.cpp` 中直接的 A 版窗口文本读写 API 已基本清空
* `lang_load_string_array()` 已被识别为 legacy-only 兼容接口，且当前无直接调用点
* 多个第三方/历史边界点已补充日志、失败保护与注释说明

## 5.2 当前仍保留的兼容边界

这些边界当前仍存在，但已被明确标记、收口并补充可观测性：

* `config.cpp`：窄字符名表兼容链路（已集中到少数辅助转换函数）
* `output_asio.cpp`：ASIO 驱动/设备名边界（已集中到少数转换辅助函数）
* `synthesizer_vst.cpp`：VST 插件名边界（已集中到少数转换辅助函数）
* `asio/asiolist.cpp`：注册表 CLSID 文本边界
* `language.cpp`：legacy-only Win32 string resource 接口（已在声明、实现与日志中明确标记）

## 5.3 当前暂不继续硬改的原因

* 一部分问题属于第三方 API / 注册表 / 历史脚本解析边界
* 继续硬改将明显增加回归风险
* 现阶段更合适的策略是：
  * 先明确边界
  * 先补日志与兜底
  * 在后续有明确收益时再做结构性替换
